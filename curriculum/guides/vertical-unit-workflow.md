# Vertical teaching-unit production workflow

Produce one defensible end-to-end teaching unit before multiplying chapters,
slides, or labs horizontally. The unit is a coherent arc from claim and authentic
question through learner evidence, physical verification, bilingual delivery,
pilot correction, and controlled release.

## Entry contract

Before work starts, freeze a unit brief with semantic ID, audience/time budget,
outcomes and progression levels, prerequisites/downstream dependencies, real-life
question/context layer, misconceptions, safety boundary, personal evidence,
required §13.1 resources/applicability decisions, apparatus paths, owners, and
release target. Limit work in progress to the capacity of technical review,
translation/adaptation, physical pilot, accessibility review, and defect closure.

## Eight-step loop

| Step | Work and required artifacts | Exit evidence |
|---|---|---|
| 1. Brief and retrieve | Freeze IDs, outcomes/prerequisites/downstream use, authentic requirement-changing question, misconceptions, workload, safety/context/access scope, evidence and resource applicability. | Approved brief and applicability inventory; no silent omission. |
| 2. Establish accuracy | Create claim register; classify claims; independently derive/check calculations; identify primary sources, versions, limits, and review triggers. | Major claims have verification plans; unsafe/unsupported premise stops authoring. |
| 3. Design evidence first | Draft exit check, practical/design evidence, rubric rows/cut, required evidence-ladder rung, individual/team boundary, transfer and remediation. | Evidence samples the intended outcome at the stated independence, safety, and reasoning level. |
| 4. Freeze shared technical contract | Equations/notation, values, figures, data/code, equipment capabilities, expected ranges, fault/anomaly, design decision, acceptance test, parity fields. | Technical contract version is reviewable and shared by EN/FR. |
| 5. Author one learning arc | Natural EN/FR chapter/explanation, professor plan/notes, slides, demo/exploration/lab, problem set/solution, items/assessment, accessible/print/offline alternatives. | Discover → Understand → Experiment → Analyze → Design is coherent; package inventory linked. |
| 6. Verify independently | Recompute, compile, simulate, run code/HDL/PCB hooks, inspect citations/originality/licenses, safety/access/parity/build review, physically test every apparatus/substitution path. | Independent evidence, ranges/raw data, facet decisions, zero unresolved critical defect. |
| 7. Pilot teaching | Second instructor rehearses; learners from intended EN/FR pathways complete formative usability pilot; later a broader small cohort exercises declared access/equipment paths. | Timing, prerequisite, language, engagement, cognitive load, misconception, apparatus and access defects are dispositioned. |
| 8. Correct and release | Correct shared and paired assets, rerun affected checks/pilots, approve both languages, tag/archive offline outputs, publish limitations/review dates, feed evidence to next unit. | QA/release decision and audit event under the controlled lifecycle. |

## Required vertical trace

For every outcome, a reviewer must traverse:

```text
prerequisite → authentic question/prediction → model/explanation
→ personal learner action → raw physical/data evidence → anomaly/debug
→ design decision → acceptance test → assessment/rubric
→ remediation/transfer → downstream dependency
```

Broken traces include an assessed outcome never taught, a lab with no individual
evidence, a context with no changed requirement, a claim with no source/test, a
solution inconsistent with the learner version, or a slide-only “course.”

## Handoff packet at each review boundary

Provide immutable source version, manifest/package inventory, applicability
decisions, technical contract, claim register, artifact metadata/provenance, EN/FR
pair list, apparatus profiles and pilot data, safety/access notes, rubric/answers
under correct access, build/offline instructions, defect log, and requested facet/
lifecycle decision. Reviewers must not reconstruct missing provenance from chat or
memory.

## WIP and defect policy

- Do not open another large unit when review, adaptation, lab, or pilot queues
  contain aging critical/high defects or unreviewed vertical slices.
- A defect in shared meaning opens both language editions. Rework the earliest
  invalidated step; do not patch the final PDF alone.
- Reuse verified structures and shared assets, but never inherit approval for a
  changed claim, apparatus, fault, context, assessment, or translation.
- Capture actual author/reviewer/pilot hours and defects so estimates improve for
  the next unit.

## Definition of a usable handoff

A new qualified author or instructor can locate the authoritative source, explain
the pedagogical decisions, build both languages offline, set up each declared path,
predict its ranges, recover from the seeded fault, administer/score evidence,
route remediation, and identify known limitations without undocumented coaching.
